1 Purpose

This specification defines the canonicalization, hashing, conformance, proposal, planning, admission, refusal, receipt, bundle, and evidence identity rules required for reproducible SERA implementations.

Its purpose is to remove implementation freedom from the parts of the SERA pipeline that must agree across independent implementations.

The governing conformance law is:

`same declared input + same frozen profiles -> same declared result`

For canonical identities:

`same canonical subject -> same canonical bytes -> same SHA-256`

For deterministic resolution:

`same canonical document + same schema + same rules + same scope + same anchor -> same resolution state`

For canonical planning:

`same canonical document + same resolved target + same action + same parameters -> same canonical plan`

The preferred architecture notation is:

`E_c = plan(E_s, D)`

Planning notation is deliberately distinct from the Shunyaya collapse operator:

`phi((m, a)) = m`

because:

`planning constructs an edit artifact`

while:

`collapse preserves a declared value`

For admission evidence:

`same target + same declared dependencies + same plan + same mutation envelope -> same admission identities`

For preview commitment:

`same committed target + same admission subject + same action + same canonical plan + same envelope -> same preview commitment`

For refusal evidence:

`same refusal subject -> same refusal witness hash`

For receipt evidence:

`same receipt subject + same previous receipt identity -> same receipt hash`

For replay:

`same genesis + same valid receipt chain + same frozen profiles -> same reconstructed final document identity`

This specification defines the identity and conformance boundary required for:

- canonical document text;
- canonical JSON;
- SHA-256 subjects;
- proposal canonicalization;
- strict machine proposal handling;
- target fingerprints;
- candidate-set hashes;
- canonical plans;
- admission dependency fingerprints;
- mutation envelopes;
- mutation-envelope hashes;
- preview commitments;
- the No Silent Retarget Law;
- structural edit permits;
- refusal witnesses;
- receipt hashes;
- evidence-bundle fields;
- grapheme-aware profile direction;
- refusal and conformance behavior;
- independent replay verification;
- differential resolver comparison;
- bounded commutativity evidence.

This specification preserves the historical v0.8.1 profile identities while defining the current v0.9.2 conformance boundary.

The v0.9.2 reference implementation now uses the current proposal, strict-agent, Preview v2, refusal-witness, Receipt v2, Session Bundle v2, and frozen grapheme-aware profiles defined by this specification.

Where a newer profile changes a canonical field name or hash subject, the old profile remains valid for evidence created under that old profile.

The governing migration law is:

`normative identity change -> new profile identity`

not:

`normative identity change -> silent reinterpretation of old evidence`

This specification does not establish independent verification by itself.

A conforming implementation may still share a specification error with another implementation.

Independent verification requires a separate implementation path that reconstructs declared evidence rather than trusting producer-generated PASS labels.

3 Normative Language

The words MUST, MUST NOT, REQUIRED, SHALL, SHALL NOT, SHOULD, SHOULD NOT, RECOMMENDED, MAY, and OPTIONAL are normative.

A conforming implementation MUST distinguish:

- architectural requirements;
- profile-specific requirements;
- current-reference behavior;
- future profile extensions.

A change to a normative canonical identity rule requires a new applicable profile or specification version.


4 Canonical Byte Encoding

4.1 Character Encoding

Canonical text and canonical JSON MUST be hashed as UTF-8 bytes.

A canonical hash subject is:

`SHA256(UTF8(canonical_subject))`

SHA-256 output MUST be represented as:

- lowercase hexadecimal;
- exactly 64 hexadecimal characters;
- no prefix;
- no spaces;
- no line break included unless the canonical subject itself contains one.

Example empty-string SHA-256:

`e3b0c44298fc1c149afbf4c8996fb92427ae41e4649b934ca495991b7852b855`


4.2 Byte-Order Mark

The current canonical `normalizeText` profile does not silently remove a Unicode byte-order mark that has already entered the decoded string.

Therefore:

`U+FEFF in decoded content -> content unless a higher-level adapter removes it before canonicalization`

An adapter that removes a byte-order mark MUST do so before the SERA canonical text boundary and MUST declare that preprocessing behavior.

A future stricter input profile MAY define mandatory BOM rejection or removal.


5 Canonical Document Text

5.1 Normalization Function

For the current reference family, canonical document text is:

`normalizeText(text) = NFC(lineEndingNormalize(text))`

where:

`CRLF -> LF`

`CR -> LF`

and all other characters are preserved.

The order is:

`decoded text -> line-ending normalization -> Unicode NFC normalization`

Equivalent implementations MAY apply NFC before line-ending normalization if and only if the resulting canonical text is identical for every accepted input.


5.2 Line Endings

The following input line endings normalize to LF:

`0x0D 0x0A -> 0x0A`

`0x0D -> 0x0A`

LF remains LF.

Canonical document text therefore uses LF internally.


5.3 Unicode Normalization

The current canonical normalization form is:

`NFC`

Two canonically equivalent Unicode sequences MUST produce the same canonical document text after NFC normalization.


5.4 Whitespace Preservation

Canonical document normalization MUST NOT otherwise:

- trim leading whitespace;
- trim trailing whitespace;
- collapse repeated spaces;
- collapse repeated newlines;
- change tabs;
- normalize punctuation;
- change letter case.

Whitespace may be structurally significant.

The law is:

`canonicalization normalizes representation, not document meaning`


5.5 Document Hash

The canonical document hash is:

`doc_hash = SHA256(UTF8(normalizeText(document)))`

The document hash represents the complete canonical document.

It is distinct from:

- target fingerprint;
- admission fingerprint;
- plan hash;
- envelope hash;
- preview commitment;
- permit ID;
- receipt hash.

These identities MUST NOT be treated as interchangeable.


6 Canonical JSON

6.1 Purpose

Canonical JSON is used to construct deterministic hash subjects for structured SERA evidence.

The canonical JSON function MUST produce the same JSON string for equivalent supported objects regardless of source object insertion order.


6.2 Object Key Ordering

Object keys MUST be sorted in ascending lexical order before serialization.

All normative SERA field names in the current profiles are ASCII.

For ASCII field names:

`ascending lexical order = ascending UTF-8 byte order = ECMAScript default string order`

An implementation MUST NOT depend on source map or object insertion order.


6.3 Recursive Ordering

Object-key sorting is recursive.

Every nested object MUST be canonicalized independently.

Example source:

`{"z":1,"a":{"y":2,"b":3}}`

Canonical form:

`{"a":{"b":3,"y":2},"z":1}`


6.4 Array Ordering

Array order MUST be preserved exactly.

A conforming implementation MUST NOT sort arrays unless the applicable profile explicitly defines the array as a mathematical set with a separate canonical ordering rule.


6.5 Supported Scalar Types

Canonical hash-domain values MAY contain:

- string;
- integer;
- boolean;
- null;
- array;
- object.

Normative numeric fields SHOULD use integers within the exact interoperable range:

`-(2^53 - 1) <= n <= 2^53 - 1`

The following values MUST NOT appear in canonical SERA JSON:

- NaN;
- positive infinity;
- negative infinity;
- undefined;
- functions;
- runtime object references.

Negative zero MUST serialize as:

`0`


6.6 Whitespace

Canonical JSON MUST contain no insignificant whitespace.

No spaces, indentation, or line breaks may be inserted outside JSON string values.

Example:

`{"a":1,"b":2}`

not:

`{ "a": 1, "b": 2 }`


6.7 String Escaping

Strings MUST use valid JSON string escaping.

The canonical serializer MUST preserve Unicode scalar content while escaping characters required by JSON syntax.

Canonical JSON is then encoded as UTF-8 before hashing.


6.8 Canonical JSON Hash

For any supported structured subject `X`:

`hashValue(X) = SHA256(UTF8(canonicalJSON(X)))`


6.9 Local JSON Schema Validation

The current repository may provide local JSON Schema files corresponding to declared SERA evidence objects.

The current companion files are:

`SERA_Proposal_Schema_v1_0.json`

`SERA_Refusal_Witness_Schema_v1_0.json`

`SERA_Receipt_Schema_v2_0.json`

`SERA_Session_Bundle_Schema_v2_0.json`

These files validate structural shape.

They do not replace the canonical rules in this specification.

The governing separation is:

`JSON Schema -> structural validity`

`this specification -> normative canonical and semantic validity`

`independent verifier -> reconstructed evidence validity`

A schema validator alone cannot establish:

- receipt-chain continuity;
- correct receipt hashes;
- refusal non-mutation;
- canonical plan replay;
- mutation-envelope containment;
- No Silent Retarget;
- final reconstructed document identity.

The schema files require no hosted SERA service.

The optional JSON Schema dialect declaration does not make network retrieval part of SERA semantics or admission authority.


7 Offset and Character Semantics

7.1 Current Offset Unit

The current browser reference declares:

`offset_unit = UTF16_CODE_UNIT`

All spans and positions used by the current reference plan representation are expressed in UTF-16 code units.

A span:

`[start, end]`

uses:

`start = inclusive`

`end = exclusive`


7.2 Structural Character Units

The current v0.9.2 reference implementation exposes two distinct narrow text scopes:

`CODE_POINT`

`GRAPHEME`

The evidence coordinate representation remains:

`offset_unit = UTF16_CODE_UNIT`

The `CODE_POINT` scope resolves Unicode code points.

The `GRAPHEME` scope resolves extended grapheme clusters under:

`SERA-UNICODE-GRAPHEME-1-D01`

Therefore:

`UTF16_CODE_UNIT != UNICODE_CODE_POINT`

and:

`UNICODE_CODE_POINT != EXTENDED_GRAPHEME_CLUSTER`


7.3 Grapheme Clusters

The current human-facing grapheme profile is:

`SERA-UNICODE-GRAPHEME-1-D01`

It declares:

`unicode_version = 17.0.0`

`segmentation_specification = UAX_29`

`segmentation_revision = 47`

`grapheme_mode = EXTENDED_GRAPHEME_CLUSTER`

`tailoring = NONE`

A user-perceived character may contain multiple Unicode code points.

A conforming implementation MUST NOT silently relabel `CODE_POINT` as `GRAPHEME`.

Any later Unicode version, segmentation revision, or tailoring rule that changes normative segmentation behavior requires a new applicable profile identity.


7.4 Cross-Implementation Requirement

An implementation using another native offset unit MUST convert to the declared canonical offset unit before producing canonical SERA evidence.

For the current reference family:

`canonical evidence offset unit = UTF16_CODE_UNIT`


7.5 Canonical Text Before Structural Segmentation

Unless an applicable profile explicitly declares another boundary, structural segmentation operates on canonical document text.

For the current family:

`decoded text -> LF normalization -> NFC normalization -> structural segmentation`

An implementation MUST NOT produce canonical evidence from offsets calculated against a different text representation without a declared conversion.


7.6 Offset and Structural Unit Separation

The evidence offset unit and the human-facing structural unit are independent declarations.

The current v0.9.2 grapheme profile validly declares:

`offset_unit = UTF16_CODE_UNIT`

and:

`character_unit = EXTENDED_GRAPHEME_CLUSTER`

This means:

`evidence coordinates use UTF-16 code units`

while:

`character navigation resolves extended grapheme clusters`

The two concepts MUST NOT be treated as interchangeable.


7.7 Host Coordinate Translation Boundary

The current canonical evidence offset unit is:

`UTF16_CODE_UNIT`

A host adapter using another native coordinate unit MUST translate at the declared adapter boundary before producing current-profile canonical SERA coordinates.

The governing distinction is:

`canonical structural identity != host-native coordinate representation`

A UTF-8 byte offset, Unicode code-point index, grapheme index, or other host-native coordinate MUST NOT be consumed as a `UTF16_CODE_UNIT` position without a declared deterministic conversion.

Cross-host conformance should verify:

`canonical structural span -> translated host span -> same intended text`


8 Schema and Profile Identity

Every conformance subject that depends on a schema or rule family MUST bind the applicable identity.

Current browser schema identities include:

TEXT:

`schema_id = writing.basic`

`schema_version = 1.1.0`

MARKDOWN:

`schema_id = markdown.basic`

`schema_version = 1.1.0`

Current primary rules identity:

`rules_version = SK-RULES-1.1.0`

A conforming implementation MUST NOT silently substitute another rule profile while preserving the same declared identity.

`changed normative rule -> changed applicable profile identity`


8.1 Current and Defined Profiles

This specification distinguishes:

`current implemented profile`

from:

`defined but not currently demonstrated profile`

The evidence produced by an implementation MUST bind the profile actually used.

An implementation MUST NOT claim conformance to a profile merely because the profile is described in this document.

For the current v0.9.2 reference, the implemented profile set includes Proposal v1, Strict Agent v1, Preview v2, Refusal Witness v1, Receipt v2, Session Bundle v2, and the frozen Grapheme v1 profile.

`SERA-COMMUTE-1-D01` remains defined but not currently demonstrated.

`SERA-RECEIPT-1`

`SERA-SESSION-BUNDLE-1`

remains interpreted under those identities.

A verifier supporting those profiles MUST reconstruct their original canonical subjects.

It MUST NOT rewrite historical fields into the new profile and then compare the resulting hashes as if they were unchanged.

9 Resolution Conformance

9.1 Inputs

A resolution conformance subject is determined by:

- canonical document text;
- schema identity;
- schema version;
- rules version;
- scope;
- anchor;
- any explicit profile parameters.

The general relation is:

`resolve(canonical_document, schema, rules, scope, anchor) -> declared resolution`


9.2 Resolution States

The supported top-level states are:

`RESOLVED`

`AMBIGUOUS`

`UNSUPPORTED`


9.3 RESOLVED

A resolved result MUST identify one target.

The target MUST contain enough information to reproduce its identity under the active profile.

For the current reference family:

- span;
- structural path;
- fingerprint.


9.4 AMBIGUOUS

An ambiguous result MUST preserve the candidate order defined by the resolver profile.

Candidate order MUST NOT depend on:

- object iteration order;
- runtime hash-table order;
- wall-clock timing;
- random choice.

A candidate set MAY carry a candidate-set hash.


9.5 UNSUPPORTED

An unsupported result MUST provide a stable reason code under the applicable profile.

A conforming implementation MUST NOT silently convert an unsupported result into an arbitrary resolved target.


10 Target Fingerprint Profile

10.1 Profile

Current target fingerprint profile:

`SK-FP-HYBRID-1-D01`


10.2 Purpose

The target fingerprint binds:

- target structural scope;
- target structural path;
- target canonical content;
- bounded left context;
- bounded right context;
- canonical offset profile identity.

It is intended to identify structural target meaning more strongly than raw offsets alone.


10.3 Context Window

The current context window is:

`40 UTF-16 code units`

For target span:

`[start, end]`

left context is:

`text[max(0, start - 40) : start]`

right context is:

`text[end : min(document_length, end + 40)]`


10.4 Target Text Hash

`text_hash = SHA256(UTF8(normalizeText(target_text)))`


10.5 Context Hashes

`before_hash = SHA256(UTF8(normalizeText(before_context)))`

`after_hash = SHA256(UTF8(normalizeText(after_context)))`


10.6 Fingerprint Payload

The payload is:

`profile`

`offset_unit`

`scope`

`path`

`text_hash`

`context_window`

`context.before_hash`

`context.after_hash`

Conceptually:

{
  "profile": "SK-FP-HYBRID-1-D01",
  "offset_unit": "UTF16_CODE_UNIT",
  "scope": "<scope>",
  "path": [...],
  "text_hash": "<sha256>",
  "context_window": 40,
  "context": {
    "before_hash": "<sha256>",
    "after_hash": "<sha256>"
  }
}

The target fingerprint is:

`target_fingerprint = SHA256(UTF8(canonicalJSON(payload)))`


10.7 Structural Path

The structural path is an ordered integer array.

Its exact interpretation is schema-specific.

Examples may include:

`[paragraph_index, sentence_index]`

or:

`[unit_index]`

The path order MUST be preserved.


10.8 Target Span Exclusion

The current fingerprint payload does not directly include raw `start` and `end` span offsets.

The fingerprint derives identity from:

- path;
- scope;
- target text;
- bounded context.

The resolved result may still carry the span separately.

This distinction is intentional:

`span = execution coordinate`

`fingerprint = structural target identity evidence`


11 Candidate-Set Hash

For an ambiguous result, the current candidate-set hash payload is an ordered array.

For each candidate:

`index`

`span`

`path`

`fingerprint`

The candidate-set hash is:

`candidate_set_hash = SHA256(UTF8(canonicalJSON(candidate_array)))`

Candidate order MUST remain exactly the resolver-defined order.

Reordering candidates changes the candidate-set hash.


12 Canonical Plan Representation

12.1 Purpose

The canonical plan is the exact classical edit representation committed for admission.

The current plan is an ordered array of canonical operations.

Operation order is significant.

The preferred architecture notation is:

`E_c = plan(E_s, D)`

where:

`E_s = structural edit intent`

`D = canonical document state`

`E_c = canonical classical edit plan`

The notation `plan()` MUST be used for forward SERA planning documentation.

The symbol `phi` is reserved in the wider Shunyaya ecosystem for collapse relations such as:

`phi((m, a)) = m`

The two meanings MUST NOT be conflated.


12.2 Supported Current Operations

Current canonical operations include:

`DELETE_RANGE`

`INSERT_AT`

`REPLACE_RANGE`

A future profile MAY add:

`MOVE_RANGE`

or other operations only with a new compatible normative definition.


12.3 DELETE_RANGE

Canonical fields:

`op`

`start`

`end`

Example:

{
  "op": "DELETE_RANGE",
  "start": 12,
  "end": 28
}


12.4 INSERT_AT

Canonical fields:

`op`

`position`

`content`

Example:

{
  "op": "INSERT_AT",
  "position": 28,
  "content": "Inserted text"
}


12.5 REPLACE_RANGE

Canonical fields:

`op`

`start`

`end`

`content`

Example:

{
  "op": "REPLACE_RANGE",
  "start": 12,
  "end": 28,
  "content": "Replacement"
}


12.6 Non-Canonical Operational Metadata

The current reference planner may temporarily carry operational metadata such as:

`move_meta`

`paste_meta`

These fields MUST be removed from the canonical plan before plan hashing and preview commitment.

The current canonicalization law is:

`canonical plan = raw plan - non-canonical transient operation metadata`


12.7 Plan Hash

`plan_hash = SHA256(UTF8(canonicalJSON(canonical_plan)))`


12.8 Plan Determinism

Under one planning profile:

`same canonical document + same target + same action + same parameters -> same canonical plan`

A changed plan hash after preview is a staleness condition under the current admission model.


12.9 Structural-to-Classical Plan Equivalence

Where an applicable profile claims structural-to-classical equivalence:

`apply_structural(E_s, D) = apply_classical(plan(E_s, D), D)`

The equivalence claim MUST be bounded to:

- one declared schema;
- one declared rules profile;
- one declared planning profile;
- one declared action family.

A conformance vector MAY additionally bind the deterministic post-action anchor.


13 Admission Dependency Payload

13.1 Purpose

The admission dependency payload identifies the exact structure that must remain equivalent for a proposal to keep the same admitted meaning.

The current profile is:

`SERA-ADMISSION-DEP-1-D01`


13.2 Common Fields

Every current admission dependency payload contains:

`profile`

`schema_id`

`schema_version`

`rules_version`

`scope`

`action`

`target_fingerprint`


13.3 DELETE and CUT Dependencies

Additional fields:

`delete_range`

`delete_preimage_hash`

The preimage hash is:

`SHA256(UTF8(exact canonical document substring in delete_range))`

Because the document entering planning is already canonical text, the exact substring is the hash subject.


13.4 DUPLICATE Dependencies

Additional fields:

`insertion_point`

`insertion_context_hash`

The current insertion context window is:

`24 UTF-16 code units before insertion point`

plus:

`24 UTF-16 code units after insertion point`

clamped to document boundaries.

The context hash subject is the exact canonical document substring in that window.


13.5 PASTE Dependencies

Additional fields:

`insertion_point`

`insertion_context_hash`

`clipboard_content_hash`

`clipboard_scope`

The clipboard content presented to the planning layer MUST have a declared normalization boundary.

For conformance with the current reference path, adapter-provided clipboard text SHOULD enter the planner in canonical NFC/LF form.


13.6 MOVE Dependencies

For:

`MOVE_BACK`

`MOVE_FORWARD`

additional current fields are:

`relation`

`relation_range`

`relation_preimage_hash`

The relation preimage covers the exact canonical source-to-destination relation range used by the current plan.


13.7 Admission Fingerprint

`admission_fingerprint = SHA256(UTF8(canonicalJSON(dependency_payload)))`


13.8 Dependency Meaning

The admission fingerprint is not the whole-document hash.

Therefore:

`document_hash changed`

does not automatically imply:

`admission_fingerprint changed`

This allows a structurally unrelated edit to preserve another proposal's admission identity.


14 Mutation Envelope

14.1 Profile

Current profile:

`SERA-MUTATION-ENVELOPE-1-D01`


14.2 Purpose

The mutation envelope bounds the write authority of the admitted canonical plan.

The governing law is:

`write_set(plan) subseteq admitted_mutation_envelope`


14.3 Envelope Object

The current envelope contains:

`profile`

`operations`

The operation list corresponds positionally to the canonical plan operation list.


14.4 INSERTION_POINT

For `INSERT_AT`:

{
  "kind": "INSERTION_POINT",
  "position": <position>
}


14.5 RANGE

For `DELETE_RANGE` or `REPLACE_RANGE`:

{
  "kind": "RANGE",
  "start": <start>,
  "end": <end>
}


14.6 MOVE

A future canonical `MOVE_RANGE` operation may use:

{
  "kind": "MOVE",
  "start": <start>,
  "end": <end>,
  "destination": <destination>
}

The current browser planner represents structural movement as `REPLACE_RANGE`, so the current canonical envelope for that operation is `RANGE`.


14.7 Envelope Hash

`mutation_envelope_hash = SHA256(UTF8(canonicalJSON(envelope)))`


14.8 Positional Validation

Under the current profile:

`MUTATION_ENVELOPE_VIOLATION`

17 Commit-Time Revalidation and Refusal Precedence

17.1 Purpose

When multiple failure conditions could apply, implementations must evaluate the current admission path in a declared order.


17.2 Current Admission Evaluation Order

The current reference family evaluates:

1. resolved snapshot exists;
2. snapshot target revalidation;
3. expected action contract construction from the resolution snapshot;
4. current action contract construction from the current canonical document;
5. admission fingerprint comparison;
6. canonical plan hash comparison;
7. mutation-envelope validation;
8. ADMITTED result.

The first applicable refusal under this order governs the result.

`CONFLICT`

`CONFLICT`

17.13 Successful Admission

Only after all preceding checks pass:

`ADMITTED`


17.14 No Silent Retarget Law

The target represented by the active preview commitment MUST remain the target admitted at commit time.

The governing law is:

`preview_target_fingerprint != commit_target_fingerprint -> no execution under old commitment`

A valid continuation requires:

`re-resolve -> new preview -> new commitment -> new admission`

An implementation MUST NOT:

- preserve the old preview commitment;
- silently resolve a different target;
- execute against the new target under the old commitment.

A verifier that detects this condition SHOULD report:

`TARGET_RETARGETED_WITHOUT_NEW_COMMITMENT`

The top-level SERA admission state remains determined by the applicable profile's refusal ordering.

For the current family, a changed target fingerprint normally produces:

`STALE`

with reason:

`TARGET_STALE`

The No Silent Retarget Law is therefore an invariant over the full proposal-to-commit path, not a replacement top-level state.


18 Receipt Canonicalization

18.1 Legacy v1 and Current v2 Profiles

`SERA-RECEIPT-1`

The current v0.9.2 receipt profile is:

`SERA-RECEIPT-2`

The current reference implementation applies the deterministic sequence, previous-link, `ts = null`, and canonical receipt-hash rules defined by the v2 profile in Section 40.


18.2 Receipt Sequence

Each receipt contains:

`seq`

The first receipt uses:

`seq = 0`

Subsequent receipts increment by exactly one.


18.3 Previous Receipt Link

The first receipt contains:

`prev = null`

Every later receipt contains:

`prev = previous_receipt.receipt_hash`


18.4 Timestamp Field

The current deterministic receipt profile uses:

`ts = null`

Wall-clock time is therefore not part of receipt identity.


18.5 Receipt Version

Each v1 receipt contains:

`v = SERA-RECEIPT-1`


18.6 Receipt Hash Construction

Before hashing, the receipt contains all canonical receipt fields except:

`receipt_hash`

Then:

`receipt_hash = SHA256(UTF8(canonicalJSON(receipt_without_receipt_hash)))`

The computed hash is appended as:

`receipt_hash`


18.7 Chain Verification

For every receipt:

`recomputed receipt_hash = stored receipt_hash`

For every receipt after the first:

`receipt.prev = previous.receipt_hash`

A mismatch invalidates the chain result.


18.8 Forward Receipt Profile

The forward receipt profile is:

`SERA-RECEIPT-2`

It preserves deterministic chaining while adding verifier-facing requirements for:

- canonical proposal identity where present;
- canonical plan naming;
- refusal witness identity;
- explicit profile binding.

The exact forward rules are defined in Section 39.

33 Baseline Conformance Laws

The primary laws of this specification are:

`same canonical document -> same document hash`

`same supported canonical object -> same canonical JSON -> same hash`

`same target subject -> same target fingerprint`

`same ordered candidate set -> same candidate-set hash`

`same structural action subject -> same canonical plan -> same plan hash`

`same declared admission dependencies -> same admission fingerprint`

`same canonical plan -> same mutation envelope under the same profile`

`same preview subject -> same preview commitment`

`same permit subject -> same permit ID`

`same receipt subject + same previous receipt hash -> same receipt hash`

`same frozen vector + same frozen profiles -> same declared result`

The governing principle is:

`canonical identity must come from declared bytes, not implementation accident`

34 Verifier-Facing Profile Set and Transition

34.1 Purpose

This section defines the verifier-facing profile set adopted by the SERA v0.9.2 reference implementation and the remaining transition rules required before independent verification is frozen.

The objective is to prevent a verifier from institutionalizing terminology or canonical subjects that are no longer current.

The governing transition law is:

`freeze semantics -> freeze vectors -> implement independent verifier`

The v0.9.1 profile set does not invalidate v1 evidence.

The v2 profiles create distinct canonical subjects for new evidence.


34.2 Planning Terminology

The preferred planning relation is:

`E_c = plan(E_s, D)`

The symbol:

`phi`

is reserved for Shunyaya collapse semantics.

A field or document that used `collapse_plan` under an old frozen profile remains valid under that old profile.

New verifier-facing profiles MUST use:

`canonical_plan`

where the canonical plan itself is embedded.


34.3 Profile Migration

A producer adopting the current verifier-facing profile set MUST update the declared profile identities.

It MUST NOT produce a v2 canonical subject while declaring a v1 profile identity.

The governing law is:

`new canonical subject + old profile identity -> invalid evidence`


35 Canonical Proposal Profile

35.1 Profile

Current v0.9.2 profile:

`SERA-PROPOSAL-1-D01`


35.2 Purpose

The proposal profile provides a deterministic machine-readable identity for an edit proposal before target resolution.

It makes the proposal source explicit without granting that source target or mutation authority.


35.3 Canonical Proposal Fields

The canonical proposal object contains:

`profile`

`proposal_id`

`source`

`schema`

`scope`

`locator`

`action`

`payload`

`expected_profile`

`expected_document_hash`

`range_hint`

The exact object key order in source input is irrelevant because canonical JSON sorts object keys recursively.


35.4 Source

Supported declared source values for the initial profile include:

`HUMAN`

`AI_AGENT`

`API`

`ASSISTIVE_TECHNOLOGY`

An extension MAY define additional source values under a new or compatible profile rule.

The source is evidence.

`source != target authority`


35.5 Schema

The `schema` object contains:

`id`

`version`


35.6 Locator

The locator is resolver input.

A canonical locator MUST declare a `kind`.

Example:

{
  "kind": "STRUCTURAL_PATH",
  "path": [0, 0]
}

A future profile MAY add other locator kinds with explicit canonical definitions.


35.7 Action

The action object contains:

`op`

`params`

The `params` object MUST be present.

An action with no parameters uses:

`{}`


35.8 Optional Values

For this profile, semantically absent optional values in the canonical proposal object are represented as:

`null`

They MUST NOT be silently omitted if the profile declares the field as part of the canonical proposal subject.


35.9 Proposal Hash

The proposal hash is:

`proposal_hash = SHA256(UTF8(canonicalJSON(canonical_proposal)))`

The proposal ID and proposal hash are different identities.

`proposal_id = caller-facing declared identifier`

`proposal_hash = canonical proposal content identity`


35.10 Proposal Immutability

After proposal hashing, a change to any canonical proposal field produces a different proposal hash.

An implementation MUST NOT change:

- source;
- schema;
- scope;
- locator;
- action;
- payload;
- expected profile;
- expected document hash;
- range hint;

while preserving the same proposal hash.


36 Strict Machine Proposal Profile

36.1 Profile

Current v0.9.2 profile:

`SERA-AGENT-STRICT-1-D01`


36.2 Purpose

The strict machine profile prevents machine-supplied coordinates from automatically becoming mutation authority.


36.3 Range Hint Semantics

Under this profile:

`machine-supplied range = locator hint`

not:

`machine-supplied range = admitted mutation range`

A `range_hint` MAY help identify a candidate region.

The resolver MUST independently determine the supported target under the active structural profile.


36.4 Matching Range Hint

If a range hint corresponds to the independently resolved target:

`range_hint match -> may continue`

The range hint still does not replace the target fingerprint, canonical plan, admission dependencies, or mutation envelope.


36.5 Mismatching Range Hint

If a machine-supplied range hint conflicts with the independently resolved target, the implementation MUST NOT silently use the machine range as the target.

The profile MUST produce a declared non-success path.

A corpus may use the reason:

`RANGE_HINT_TARGET_MISMATCH`

The applicable top-level state depends on the resolver and admission stage.


36.6 Direct Range Authority Forbidden

A conforming strict implementation MUST reject an execution path equivalent to:

`machine range -> direct canonical plan -> mutation`

when structural resolution was required by the active proposal profile.

The required path is:

`machine proposal -> resolve -> plan -> preview -> revalidate -> admit -> bounded execution`


36.7 Compatibility Adapters

A compatibility adapter MAY accept a legacy range-bearing request.

It MUST declare the adapter behavior.

The recommended compatibility relation is:

`legacy range -> range_hint`

The adapter MUST NOT silently convert:

`legacy range`

into:

`admitted mutation authority`

`target changed -> old preview commitment cannot authorize execution`


38 No Silent Retarget Conformance

38.1 Governing Law

`preview_target_fingerprint != commit_target_fingerprint -> no execution under old commitment`


38.2 Valid Continuation

A proposal may continue only through:

`re-resolve -> new preview -> new commitment -> new admission`


38.3 Invalid Continuation

The following path is non-conforming:

`old preview commitment + newly resolved different target -> execution`


38.4 Verification

Where the receipt or bundle provides sufficient information, an independent verifier SHOULD compare:

`preview target fingerprint`

with:

`commit target fingerprint`

A mismatch without a new preview commitment is:

`TARGET_RETARGETED_WITHOUT_NEW_COMMITMENT`


38.5 Refusal State

The No Silent Retarget invariant does not create a new top-level SERA state.

The top-level state remains determined by the applicable profile.

For the current admission family, a changed target fingerprint ordinarily produces:

`STALE`

with:

`TARGET_STALE`

`unrelated earlier change`

`target content change`

`target boundary change`

`required separator change`

`required destination relation change`

`mutation-envelope escape`


44.3 Expected Pattern

Example expected outcomes:

`unrelated earlier change -> may remain ADMITTED`

`unrelated later change -> may remain ADMITTED`

`target content change -> STALE`

`target boundary change -> STALE`

`required destination relation change -> STALE OR CONFLICT`

`mutation-envelope escape -> CONFLICT`


44.4 Conformance Value

The matrix distinguishes:

`whole document changed`

from:

`declared proposal dependency changed`

`IRRELEVANT`

`DEPENDENCY_RELEVANT`

`TARGET_MUTATING`

`RULES_RELEVANT`

`SCHEMA_RELEVANT`

`ADMITTED`

`STALE`

or:

`CONFLICT`

`pre_document_hash != current_document_hash`

while:

`required target and admission dependencies remain valid`

and the proposal revalidates to:

`ADMITTED`

This is Structural Admission Survival.

The conformance meaning is:

`whole-document identity change alone is not sufficient evidence of staleness under a dependency-aware admission profile`

49 Final Conformance Laws

The primary v0.2.0 laws are:

`same canonical document -> same document hash`

`same supported canonical object -> same canonical JSON -> same hash`

`same canonical proposal -> same proposal hash`

`proposal source != target authority`

`machine range under strict profile -> locator hint, not mutation authority`

`same target subject -> same target fingerprint`

`same ordered candidate set -> same candidate-set hash`

`same structural action subject -> same canonical plan -> same plan hash`

`same declared admission dependencies -> same admission fingerprint`

`same canonical plan -> same mutation envelope under the same profile`

`same v2 preview subject -> same v2 preview commitment`

`preview target changed -> old commitment cannot execute`

`same permit subject -> same permit ID`

`same refusal subject -> same refusal witness hash`

`refusal -> no successful mutation`

`same receipt subject + same previous receipt hash -> same receipt hash`

`same genesis + same valid replay evidence + same frozen profiles -> same reconstructed final identity`

`same frozen vector + same frozen profiles -> same declared result`

`profile identity determines canonical interpretation`

`new canonical subject -> new applicable profile identity`

The governing principle remains:

`canonical identity must come from declared bytes, not implementation accident`
