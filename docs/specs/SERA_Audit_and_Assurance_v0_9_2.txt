1 Purpose

This specification defines the audit, permanent-regression, feature-coverage, assurance, and release-acceptance model for the SERA browser reference implementation.

The current demonstrated implementation is:

`SERA v0.9.2`

with:

`Structural Keyboard Reference Surface v0.9.2`

under:

`SERA-AUDIT-1-D04`

The current primary release gate is:

`await SERA_AUDIT.runAll()`

The latest verified browser result is:

`329/329 PASS`

with:

`INTEGRITY REGRESSION = 0`

`UNEXPECTED MUTATION = 0`

`KNOWN REGRESSIONS = 27/27 PASS`

`FEATURE COVERAGE = 34/34 PASS`

`FINAL STATUS = PASS`

The audit system is designed to ensure that:

- core structural behavior remains stable after updates;
- every discovered deterministic bug becomes a permanent regression test;
- newly implemented deterministic features become explicit coverage obligations;
- human-facing and machine-facing workflows are tested as complete interaction flows where practical;
- refusal is treated as a successful non-mutating outcome when refusal is the correct result;
- old preview authority cannot silently transfer to a changed target;
- machine-supplied coordinates do not become direct mutation authority under the strict proposal profile;
- preview v2, receipt v2, refusal witnesses, grapheme handling, multilingual demonstrations, and verifier-fixture generation remain inside the permanent audit boundary;
- partial audits do not erase the latest complete full-audit record;
- the user's working state is restored after full audit execution;
- producer-side audit results remain clearly separated from independent verification.

The governing development law is:

`bug found -> fix implemented -> permanent regression test added -> full audit must pass`

The governing feature law is:

`implemented deterministic feature -> declared audit coverage -> permanent regression protection`

The governing release law is:

`full audit PASS + feature coverage PASS + known regressions PASS + state restoration PASS + zero unexpected mutation -> reference release candidate accepted`

3 Assurance Principles

3.1 One Principal Release Gate

The reference implementation exposes one primary release command:

`await SERA_AUDIT.runAll()`

The full audit is the authoritative producer-side release gate for the current implementation.

Focused and quick audits support development and investigation.

They do not replace the full release gate.


3.2 Bugs Become Permanent Tests

When a deterministic bug is discovered:

1. reproduce the failure;
2. fix the implementation;
3. add a permanent named regression;
4. map the regression to one or more executable tests;
5. run the complete full audit;
6. accept the update only after the complete release gate passes.

The objective is:

`discover each deterministic bug once`


3.3 Deterministic Features Require Coverage

A deterministic feature is not considered represented in the assurance boundary merely because it exists in the interface.

The required relation is:

`implemented feature -> executable tests -> feature coverage entry`

The current D04 feature registry contains:

`34 feature families`

The current verified result is:

`34/34 PASS`

with:

`UNCOVERED FEATURES = 0`


3.4 Workflow Tests Are Preferred

Where practical, tests should cover complete workflows rather than only function presence.

Examples:

`proposal -> evaluate -> resolve -> preview -> revalidate -> admit OR refuse -> evidence`

`refusal challenge open -> automatic result display -> STALE -> TARGET_STALE -> zero mutation -> refusal witness`

`multilingual demo load -> Tamil and French content present -> structural resolution remains valid -> reset path remains available`

`comparison open -> automatic live result -> Run Again -> result remains bound to current editor state`


3.5 Refusal Is a Valid Audited Outcome

A correct refusal is an assurance success when the declared rules require non-mutation.

The governing relation is:

`valid refusal -> expected state + expected reason + zero mutation + expected evidence`

The audit must fail if a refusal unexpectedly mutates the document.


3.6 No Silent Retarget Is a Permanent Invariant

The governing law is:

`preview_target_fingerprint != commit_target_fingerprint -> no execution under old commitment`

A valid continuation requires:

`re-resolve -> new preview -> new commitment -> new admission`

The audit permanently verifies that a changed target cannot inherit old preview authority.


3.7 Machine Coordinates Are Not Automatic Authority

Under:

`SERA-AGENT-STRICT-1-D01`

the audit preserves:

`machine-supplied range = locator hint`

not:

`machine-supplied range = mutation authority`

A mismatching range hint must not silently become the target.


3.8 State Preservation

The full audit follows:

`capture state -> run isolated audit -> restore state`

The captured boundary includes, where applicable:

- document text;
- schema;
- scope;
- anchor;
- clipboard state;
- receipts;
- undo stack;
- redo stack;
- proposal state;
- preview state;
- editor selection;
- editor scroll position;
- drawer and modal state;
- active inspector tab;
- current demo mode;
- relevant focus state.

The current verified result is:

`STATE RESTORATION = 1/1 PASS`


3.9 Partial Audits Do Not Replace Full Evidence

The harness separately preserves:

`last full audit`

`last quick audit`

`last focused audit`

A later quick or focused audit must not erase the complete full-audit archive.

The latest complete full-audit result is preserved separately from later partial runs.

The archive sequence number is page-session-specific and does not form part of the release identity.

The preserved result is:

`329/329 PASS`


3.10 Producer Audit and Independent Verification Remain Separate

The browser may generate:

- receipts;
- bundles;
- verifier fixtures;
- producer-side replay results;
- PASS labels.

These are not independent verification.

The separation is:

`producer audit PASS != independent verifier PASS`

Audit verdicts must not depend on:
